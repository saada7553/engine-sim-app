================================================================================
PERFORMANCE OPTIMIZATIONS — engine-simulator
================================================================================
A running log of CPU / heat optimizations to the physics simulation, why they
were made, and what is still on the table. Newest work at the top.

--------------------------------------------------------------------------------
KEY DIAGNOSIS (read this first)
--------------------------------------------------------------------------------
The hot thread is the C++ PHYSICS SIMULATION thread spawned in
  -[EngineWrapper setupEngineWithMRPath:]  ->  Simulator::simulateStep()
NOT the audio render thread and NOT the SwiftUI main thread.

Two structural facts drive everything:
1. The sim is ALWAYS ON. EngineViewModel is a @StateObject created once at the
   app root (TileSurfApp.swift), so the sim thread runs flat-out for the whole
   app session — on every tab, not just the engine view. (Decision: keep it
   always-on; the engine is meant to keep running/sounding across tabs.)
2. The sim runs in REAL TIME to feed the synthesizer. The substep count
   auto-tunes (Simulator::startFrame) to keep the audio buffer filled, so the
   thread is effectively pinned. CPU therefore scales ~linearly with the
   simulation frequency.

PROFILING GOTCHA: the Debug app links the -O0 engine-sim lib
(physics-simulation/engine-sim/build-debug, ~3.9 MB vs ~267 KB for Release).
An -O0 trace MASSIVELY over-states call overhead — e.g. atg_scs::Matrix::get
(a 1-line accessor) showed 733 ms, and GasSystem getters appear as DYLD stubs.
These collapse at -O3. ALWAYS profile the Release-linked app
(Time Profiler -> Separate by Thread + Invert Call Tree) before chasing C++
hotspots. Release flags (macOS host): -O3 -DNDEBUG -ffast-math -mcpu=apple-m1
-fvectorize -fslp-vectorize. iOS: -O3 -DNDEBUG -ffast-math. No LTO.

DONE — 2026-05-23
================================================================================

[1] Build the UI snapshot ONCE per frame, not once per substep
    File: engine-simulator/Engine/EngineWrapper.m  (the sim-thread frame loop)
    Before: the full EngineState (+ NSMutableArrays of cylinder health, the
            21-point ignition-map sampling, engine-wide health object) was
            allocated INSIDE `while (_sim->simulateStep())` — ~333x/frame at
            10 kHz / 30 fps — and all but the last iteration was discarded.
    After:  only sub-frame work stays in the loop (oscilloscope sample,
            cylinder-pressure peak tracking, money-shift event latch). The
            EngineState snapshot is assembled ONCE after the loop, reflecting
            end-of-frame state (same instant the old code's last iteration
            captured).
    Why:    eliminates ~99% of per-substep ObjC alloc/dealloc churn
            (setLatestState: -> __NSArrayM dealloc) and drops the 21-point
            getTimingAdvanceForRpm sampling from ~7,000x/frame to 21x/frame.
    Risk:   none to physics/threading; ObjC churn is real waste at any opt
            level. Peak cylinder pressure now accumulates across the whole
            frame (strictly more correct). Money-shift still polled every
            substep so no event is missed.

[2] Lower the simulation frequency 10 kHz -> 7 kHz, everywhere  (PRIMARY LEVER)
    Files: engine-simulator/Engine/EngineWrapper.m   (clamp at setup)
           engine-simulator/EngineBuilder/MRWriter.swift  (generator default)
    Change: introduced kTargetSimulationFrequencyHz = 7000 and clamp via
            MIN(engine MR value, target) at the single control point
            (_sim->setSimulationFrequency). Applies to ALL engines — existing
            MR files (which bake 10000) and newly generated — without rewriting
            MR files. MRWriter default also set to 7000 for consistency.
    Why:    sim-thread CPU scales ~linearly with frequency. 7 kHz ≈ 30% fewer
            substeps/sec -> ~30% less work on the pinned thread. Guaranteed win
            independent of profiling.
    Tunable: lower (6 kHz) = cooler but less crisp engine note + larger solver
            timestep; raise (8 kHz) if instability/audio is rough.
    COUPLED RISKS TO VERIFY ON DEVICE (not yet confirmed):
      - Audio fidelity: synth input sample rate rides on the sim frequency
        (Simulator sets synthesizer().setInputSampleRate(frequency...)), then
        resamples to 44.1 kHz. Lower rate = less bright note / more resampling
        artifacts. Listen idle -> redline.
      - Solver stability: larger dt (1/7000 s) can destabilize the rigid-body
        constraint solver at high RPM / high cylinder count. Sweep several
        engines to redline, watch for jitter/divergence.

[3] pow -> cbrt in the gas hot path
    File: physics-simulation/engine-sim/src/gas_system.cpp  (GasSystem::changeVolume)
    Change: std::pow(V + dV, 1.0/3.0)  ->  std::cbrt(V + dV)
    Why:    changeVolume runs every substep via setVolume(); std::cbrt is a
            dedicated libm routine, faster than the general pow(x, 1/3), and
            mathematically EXACT for the always-positive chamber volume. Real
            win at any optimization level (pow is a libm call, not inlined away).
    Note:   other pow() calls in gas_system.cpp are either unused dead-code
            helpers (fastPow_0_7143 / fastPow_0_2857 / fastInvSqrt) or
            setup-time only (GasSystem::flowConstant) — left untouched.

[4] Money-shift crash haptic can no longer get stuck (bugfix from [2])
    File: engine-simulator/Controls/HapticManager.swift
    Symptom: after the frequency change, the crash rumble could stay stuck on.
    Cause:   the rumble's only enforced stop was the crash-audio envelope
             dropping below kRumbleStopThreshold; the `audible` keep-alive had
             no wall-clock bound, so if the (now slower) crash dynamics let the
             envelope plateau above threshold, it ran until the 30 s Core
             Haptics event ended.
    Fix:     added kRumbleAbsoluteMaxSec = 6.0 and rumbleDeadline. The deadline
             is armed inside ensureRumbleStarted (covers every start path) and
             enforced at the top of updateMoneyshift — a single rumble session
             is force-stopped once the window elapses, regardless of audio. Not
             extended on re-fire, so the bound is strict.

================================================================================
ENGINEERING NOTES — 2D IGNITION TIMING MAP (replacing the scalar spark offset)
Recorded: 2026-05-23
================================================================================

WHY THIS CHANGE EXISTS
--------------------------------------------------------------------------------
The ECU "spark advance" map is a 2D grid (RPM × manifold load). Before this
change, editing it could only TRANSLATE the engine's timing curve up/down, never
reshape it. Reason: the C++ engine exposed a single scalar control,
IgnitionModule::m_ignitionOffset, and each tick Swift sampled the map at the
CURRENT operating point and pushed one number:

    delta = mapValue(rpm, load) - baseCurve(rpm)   ->  engine.setIgnitionOffset(delta)

The engine's real timing was therefore always `baseCurve(rpm) + scalar`, i.e. the
fixed base shape shifted by a constant. Editing a cell that wasn't the live
operating point did nothing visible; editing the live cell just slid the whole
curve. The Spark Advance scope (advance vs RPM) plotted base+scalar, so it always
looked like the same shape translated.

GOAL: make the engine run the FULL 2D map so editing a cell genuinely reshapes
spark timing per rpm AND load — while keeping community tune sharing + on-disk
persistence working unchanged.


HIGH-LEVEL DESIGN
--------------------------------------------------------------------------------
- The 2D map stays the SWIFT source of truth (EcuTuneModel). It is what gets
  persisted to EngineSpec.ecuTune and shared/downloaded via the community
  features. NONE of that format changed.
- The C++ IgnitionModule now holds a copy of the 2D table and does a bilinear
  lookup over (angular velocity, load) at simulation rate. Swift mirrors the map
  into the engine whenever it is edited (and once when an engine is wired).
- The scalar offset (m_ignitionOffset) is repurposed to carry ONLY the live
  "chaos surge" (the wobble from a deliberately-bad tune). With no chaos it is 0,
  so timing == the map exactly.
- Load is fed to the engine each tick as a single value so the lookup tracks the
  operating point at sim rate without re-marshalling the whole map.


FILES CHANGED
--------------------------------------------------------------------------------
C++ (physics-simulation/engine-sim/):
  include/ignition_module.h
    - Added #include <mutex>, <atomic>.
    - MaxMapRpmBins=16, MaxMapLoadBins=8 (fixed-size buffers; real grids are
      <=14 rpm x 6 load).
    - New API: setTimingMap(wBins,nW, loadBins,nLoad, advRad),
               clearTimingMap(), setCurrentLoad(kPa),
               getTunedAdvanceForRpm(rpm), protected sampleTimingMap(w,load).
    - Members: m_mapMutex (mutex), m_hasTimingMap, m_mapRpmCount, m_mapLoadCount,
               m_mapW[] (rad/s), m_mapLoad[] (kPa), m_mapAdv[] (rad, row-major
               [load][w]), m_mapCurrentLoad (std::atomic<double>, kPa).

  src/ignition_module.cpp
    - bracketBins() helper (ascending-bin bracket + interpolation fraction).
    - getTimingAdvance()  -> map lookup (if set) + scalar offset.  ACTUAL SPARK.
    - getTimingAdvanceForRpm() -> UNCHANGED, pure base curve. IMPORTANT: this is
      what Swift uses to SEED the ECU model and compute deviation-from-stock, so
      it must NOT return map values.
    - getTunedAdvanceForRpm() -> map-aware (used by the scope + as the displayed
      tuned curve).
    - setTimingMap()/sampleTimingMap() (bilinear). Mutex-guarded (see THREADING).

  include/engine.h
    - Pass-throughs: getTunedAdvanceForRpm, setIgnitionTimingMap,
      clearIgnitionTimingMap, setIgnitionLoad.

  src/oscilloscope_cluster.cpp
    - Spark-advance scope now sums getIgnitionOffset() + getTunedAdvanceForRpm()
      (was getTimingAdvanceForRpm()), so the scope reshapes with the map.

Obj-C++ bridge (engine-simulator/Engine/):
  EngineWrapper.h / EngineWrapper.m
    - setIgnitionTimingMapRpm:load:advancesDeg:  (NSArray<NSNumber*> in DEGREES;
      advances are row-major [load][rpm]). Bridge converts rpm->rad/s
      (units::rpm), deg->rad (kDegToRad); load stays kPa.
    - setIgnitionLoadKpa:  (live load, kPa absolute).
    - #include <vector> added.

Swift (engine-simulator/Engine/EngineViewModel.swift):
  - applyEcuTune(engine:) — per tick: setIgnitionLoadKpa(load); compute chaos
    surge; engine.setIgnitionOffset(surge) [scalar = surge ONLY];
    engine.setFuelTrim(trim); publish ignitionOffset = (mapAdvance-base)+surge.
    Does NOT marshal the whole map each tick.
  - pushIgnitionMap(engine:) — marshals ecu.rpmBins/loadBins/ignitionMap (degrees,
    row-major [load][rpm]) and calls setIgnitionTimingMapRpm:...
  - bindToEcu() — pushIgnitionMap once on wire-up, then on every map edit (the
    Combine subscription now calls pushIgnitionMap + applyEcuTune).
  - Removed the poll line `self.ignitionOffset = state.ignitionOffset`:
    ignitionOffset is now OWNED by applyEcuTune (deviation-from-stock + surge),
    because the C++ scalar only holds the surge now.

UNITS (easy to get wrong):
  - C++ timing math is in RADIANS; x-axis (rpm) is ANGULAR VELOCITY in rad/s
    (units::rpm(rpm)). getTimingAdvance samples at -crankshaft v_theta (rad/s).
  - Swift/UI work in DEGREES and RPM. Bridge does the conversion.
  - Load is kPa absolute on both sides (matches EcuTuneModel.standardLoadBins and
    manifoldPressureToAbsoluteKpa()).


WHY ignitionOffset (published) STILL MEANS "deviation from stock + surge"
--------------------------------------------------------------------------------
Two consumers read vm.ignitionOffset and must keep working:
  - OBD2CodeService (detonation/over-advance detection).
  - EcuTuningView live-cell display: baseTiming(rpm) + ignitionOffset.
Since the map now provides the base shape, the C++ scalar is surge-only. So Swift
computes the display value itself: (mapAdvance - baseAdvance) + surge. Both
consumers stay correct with no change.


COMMUNITY SHARING + PERSISTENCE — VERIFIED INTACT
--------------------------------------------------------------------------------
- EcuTune (rpmBins/loadBins/ignitionMap/fuelMap) and EngineSpec.ecuTune are
  unchanged on disk and over the wire. export()/apply()/persistTune untouched.
- On engine load, makeEcuModel applies any saved tune, then bindToEcu pushes it
  into the running engine -> a shared/downloaded tune's SHAPE now drives physics.
- Community detail preview (EcuTuneModel.forDisplay) has no live engine; it only
  renders the heatmap, so it is unaffected.
- Untuned engines: the map is seeded == base curve (flat across load), so their
  behavior is identical to before until the user edits cells.


THREADING / DATA RACE (read this before touching the map buffers)
--------------------------------------------------------------------------------
The physics runs on a background std::thread (EngineWrapper.m: _simThread loops
startFrame/simulateStep/endFrame). IgnitionModule::getTimingAdvance() runs on THAT
thread every substep. The UI/main thread writes the map (on edits) and the load
(every tick). So there are cross-thread reader/writer accesses.

FIRST ATTEMPT (rejected as insufficient): fixed-size buffers, no locks. That only
removed the crash modes (no realloc -> no use-after-free; in-bounds indexing ->
no OOB). It did NOT remove the data race: a lookup could read a half-updated map
(mix of old/new cells) and there was no ordering guarantee between the
m_hasTimingMap flag, the counts, and the cell data. "Benign on arm64 because
aligned scalar loads/stores don't tear" is NOT the same as race-free; formally it
is UB. (NOTE: the pre-existing m_ignitionOffset / m_enabled cross-thread scalars
are the same benign-race style and were left as-is.)

CURRENT FIX:
  - std::mutex m_mapMutex guards the map buffers + counts + m_hasTimingMap.
    Reader takes a consistent snapshot in getTimingAdvance/getTunedAdvanceForRpm;
    writer holds it in setTimingMap/clearTimingMap.
  - This is cheap BECAUSE the full map is written only on edits (rare); the
    hot-path reader's critical section is a few array reads. The value that
    changes every tick (current load) stays lock-free as std::atomic<double>, so
    the per-frame path never takes the lock for it.

KNOWN TRADE-OFF: a mutex shared between the real-time-ish physics thread and the
UI thread has a theoretical priority-inversion risk (if the UI thread is
preempted mid-write, the physics thread blocks until it resumes). The write
critical section is ~96 doubles and only happens on edits, so a stall would be
sub-microsecond and vanishingly rare — acceptable for a sim.

IF YOU WANT ZERO chance of the audio/physics thread blocking, switch the map to a
lock-free scheme instead of the mutex:
  - seqlock (atomic version counter; reader retries instead of blocking), or
  - double-buffer with an atomic index/pointer swap (writer fills the inactive
    buffer, then atomically publishes it; reader acquire-loads the index once).
Keep the per-tick load as an atomic scalar either way.

================================================================================
ENGINEERING NOTES — CUTTING MAIN/UI-THREAD UTILIZATION (60-80% -> lower)
Recorded: 2026-05-23
================================================================================

SYMPTOM
--------------------------------------------------------------------------------
UI thread sat at 60-80% utilization. Two blocks of work were being done
constantly, independent of how much was actually changing on screen.

BACKGROUND — THE THREE CLOCKS (so the rates below make sense)
--------------------------------------------------------------------------------
  - C++ engine integrates at ~7 kHz (engine.cpp default 10 kHz, capped to
    kTargetSimulationFrequencyHz = 7000 in EngineWrapper.m). The physics
    std::thread wakes at 30 Hz (targetFrameRate = 30.0) and runs ~233 substeps
    per frame; fresh results land only 30x/sec.
  - Data poll -> SwiftUI state: EngineViewModel Timer at 1.0/30.0 = 30 Hz. This
    is the rate fresh numbers enter SwiftUI; observing views re-evaluate body at
    most 30x/sec.
  - Pixels: TimelineView(.animation) views (gauge needles, throttle, shift
    light, loaders) and the SceneKit 3D view are vsync-driven -> 60 Hz, or
    120 Hz on a ProMotion display. NOT poll-driven.

The waste: animated views were redrawing at up to 120 Hz even though new data
only arrives at 30 Hz.


OPTIMIZATION 1 — DEMAND-DRIVEN OSCILLOSCOPE SAMPLING
--------------------------------------------------------------------------------
PROBLEM: Every 30 Hz poll, OscilloscopeManager.sample() fetched ALL 11 scope
buffers from C++ (a call + a fresh CGPoint array allocation each) and
re-published 12 @Published arrays — EVEN ON A DASHBOARD WITH ZERO OSCILLOSCOPE
TILES OPEN. Each republish re-renders observers.

FIX: Reference-counted active-type registry on OscilloscopeManager.
  - activeTypeCounts: [EngineScopeType: Int]; registerActive/unregisterActive
    (ref-counted because two tiles can show the same scope type).
  - OscilloscopeView registers the types it draws in .onAppear and releases them
    in .onDisappear.
  - sample(from:dynoActive:) now only fetches scopes that are on screen, PLUS
    torque+power while a dyno run is active (the run-results/leaderboard capture
    reads those buffers regardless of whether a scope shows them).
  - lastUpdateTime trigger only fires when something was actually sampled, so a
    scope-free dashboard does not re-render scope observers at all.
  - No locking: the poll timer runs on the main run loop and onAppear/onDisappear
    are main-thread, so activeTypeCounts is main-only.

FILES:
  Oscilloscope/OscilloscopeManager.swift — registry + rewritten sample().
  Oscilloscope/OscilloscopeView.swift     — onAppear/onDisappear register calls.
  Engine/EngineViewModel.swift            — sample(from:dynoActive: state.dynoEnabled).

NET: a typical gauge/3D dashboard does ZERO scope work per poll instead of
11 C++ fetches + 11 allocations + 12 publishes.


OPTIMIZATION 2 — SPLIT THE STATIC GAUGE DIAL FROM THE ANIMATED NEEDLE
--------------------------------------------------------------------------------
PROBLEM: UniversalGauge wrapped its ENTIRE Canvas in TimelineView(.animation)
(up to 120 Hz) and redrew bands + every tick + every TEXT LABEL each frame,
though only the needle moves. Text rasterization in Canvas is the costly part.
The gauge also held @ObservedObject on the VM, so its body also re-ran on every
30 Hz poll.

FIX:
  - Static dial (bands/ticks/labels) moved to its own Canvas, a SIBLING OUTSIDE
    the TimelineView. Drawn once per layout instead of per frame (it never
    changes between frames).
  - Only the needle + center cap redraw inside TimelineView(.animation).
  - Numeric readout in its own small TimelineView (cheap monospaced Text).
  - Dropped @ObservedObject var engineVm -> plain `let engineVm`. The needle and
    readout read the live value off the reference each timeline tick, so the
    gauge no longer re-renders its whole body on every 30 Hz poll. Engine swaps
    still recreate the gauge via `.id(engineResetId)` in TileView, so a stale
    reference is never a concern.

FILE: Gauges/UniversalGauge.swift

NET: per-frame gauge work drops from "redraw full dial with text labels" to
"stroke two lines + a cap", times however many gauges are visible. Nothing
visual is lost — the dial is identical between frames anyway.


REMAINING LEVER (NOT YET DONE) — @Observable MIGRATION
--------------------------------------------------------------------------------
EngineViewModel is a monolithic ObservableObject with ~40 @Published props.
Because rpm changes every poll, EVERY observing view re-renders at 30 Hz —
including TileView (only needs engineId; rebuilds an AnyView tree each time) and
non-animated control panels (EngineHealthView, OBD2View, ...) whose values
rarely change.

Deployment targets are macOS 26.1 / iOS 18.6, so the @Observable macro is
available. Migrating EngineViewModel to @Observable would make SwiftUI
re-render each view only when a property it actually READS changes (TileView ->
only on engine swap; control panels -> only when their own readouts move).
Scope: ~19 observer files, only 2 bindings ($vm.throttlePosition). Worth doing
as its own reviewed step.
